# NovaJudge 开发分支操作指南

这份文档用来区分两类开发：

- 公有功能：可以公开给开源版使用，推到 public 仓库 `origin`。
- 校内功能：只给校内部署使用，推到 private 仓库 `internal`。

当前远端约定：

```bash
origin   = https://github.com/pppolf/NovaJudge.git
internal = https://github.com/pppolf/NovaJudge-internal.git
```

## 一句话规则

- 公有功能走：`origin/staging` -> 功能分支 -> PR 回 `origin/staging` -> 验证后合到 `origin/master`
- 校内功能走：`internal/master` 或 `internal/staging` -> 功能分支 -> 合到 `internal/staging` -> 验证后合到 `internal/master`
- 校内专属代码不要推到 `origin`

## 开发公有功能

适合放 public 的内容：

- 通用 OJ 功能
- 通用比赛、榜单、题库、训练、API、CCS 能力
- 不含学校名称、内网地址、私有接口、监考、外部登录适配的功能

开始开发：

```bash
git switch staging
git pull origin staging
git switch -c feature/your-public-feature
```

开发完成后验证：

```bash
npm run prisma
npm run build
```

提交并推到 public：

```bash
git add -A
git commit -m "Add your public feature"
git push -u origin feature/your-public-feature
```

然后在 GitHub 上开 PR：

- base 选 `staging`
- compare 选 `feature/your-public-feature`

测试端验证通过后，再从 `staging` 合到 `master`。

## 开发校内功能

适合只放 private 的内容：

- 外部登录
- 监考、推流、控制中心
- 西华师范大学、协会、校内比赛定制
- 内网地址、私有 API、私有环境变量
- 不适合公开的管理功能

开始开发：

```bash
git switch master
git pull internal master
git switch -c internal/your-school-feature
```

开发完成后验证：

```bash
npm run prisma
npm run build
```

提交并只推到 private：

```bash
git add -A
git commit -m "Add your internal feature"
git push -u internal internal/your-school-feature
```

然后在 GitHub private 仓库里开 PR：

- 仓库选 `pppolf/NovaJudge-internal`
- base 选 `staging`
- compare 选 `internal/your-school-feature`

校内测试端验证通过后，再合到 `internal/master`。

## 功能既可能公有又可能校内怎么办

先拆成两层：

- 通用核心：放 public，比如通用接口、通用 UI、通用配置结构。
- 校内适配：放 private，比如学校登录 provider、监控服务地址、学校文案。

推荐流程：

```bash
# 先在 public 做通用核心
git switch staging
git pull origin staging
git switch -c feature/common-core

# 完成后推 origin，合到 public staging/master
```

然后在 private 仓基于校内分支加适配：

```bash
git switch master
git pull internal master
git switch -c internal/school-adapter
```

## 推送前检查清单

推 public 前，先搜这些关键词：

```bash
rg "西华|CWNU|cwnu|cwnupaa|外部登录|EXTERNAL_AUTH|CONTROL_SERVER|RTSP|RTMP|监考|推流|内网"
```

如果命中的是源码、配置、README、`.env.example`，通常不要推 public。

推 private 前，确认远端是 `internal`：

```bash
git remote -v
git branch -vv
```

## 常用命令

查看当前在哪个分支：

```bash
git branch --show-current
```

查看改了什么：

```bash
git status --short
git diff --stat
```

查看远端：

```bash
git remote -v
```

同步 public 测试分支：

```bash
git switch staging
git pull origin staging
```

同步校内生产分支：

```bash
git switch master
git pull internal master
```

## 事故处理

如果不小心把校内代码推到了 public：

1. 立刻停止继续推送。
2. 轮换所有可能泄露的密钥、Token、内网账号和外部服务凭据。
3. 删除或回滚 public 分支里的敏感改动。
4. 如果已经进入历史记录，再考虑做历史清理，但不要把历史清理当作唯一安全措施。

如果不确定该推哪里：

- 默认不要推 public。
- 先推到 private 的临时分支。
- 确认没有校内定制后，再拆出通用部分给 public。
